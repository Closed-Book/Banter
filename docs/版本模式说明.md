# 版本模式说明：LLM 实时版 vs 演示预制 TTS 版

本项目建议把“版本”说成运行模式，而不是维护两套互相漂移的代码。GitHub 上传完整源码和 LLM 代理代码，但不上传任何个人 API key。

## 模式一：LLM 实时开发版

用途：继续开发、联调真实角色扮演、测试 prompt 和 API 契约。

推荐 `.env`：

```dotenv
VITE_USE_MOCK=false
VITE_TTS_MODE=live
STEPFUN_ROLEPLAY_API_KEY=
STEPFUN_VOICE_API_KEY=
```

说明：

- 角色扮演走本地 `/api/roleplay/chat/completions` 代理。
- TTS 走本地 `/api/voice/tts-file` 代理。
- key 只存在本地 `.env`，不进前端 bundle，不提交 GitHub。
- 模板见 `.env.llm-live.example`。

## 模式二：演示预制 TTS 版

用途：路演、录屏、现场演示。优先保证稳定，不依赖实时 TTS 生成速度、限流或网络波动。

推荐 `.env`：

```dotenv
VITE_USE_MOCK=true
VITE_TTS_MODE=prebuilt
VITE_DEMO_TTS_BASE_URL=/demo-tts
```

说明：

- 回合内容走 `shared/mock-responses/` 和前端内置 demo 流程。
- 语音从 `public/demo-tts/` 读取预制 mp3。
- 不需要 `STEPFUN_ROLEPLAY_API_KEY` 或 `STEPFUN_VOICE_API_KEY`。
- 模板见 `.env.demo-prebuilt-tts.example`。
- 预制音频文件名可用 `npm run tts:name -- npc "陈总" "台词内容"` 生成。

## 模式三：纯 UI 空跑

用途：没有任何 key 时检查页面、样式、交互。

推荐 `.env`：

```dotenv
VITE_USE_MOCK=true
VITE_TTS_MODE=off
```

说明：

- 不调用 LLM。
- 不调用 TTS。
- 适合刚 clone 项目后的第一轮验证。

## GitHub 上传规则

可以上传：

- `src/`
- `server/`
- `shared/`
- `content/`
- `docs/`
- `public/demo-tts/README.md`
- `package.json`
- `package-lock.json`
- `.env.example`
- `.env.llm-live.example`
- `.env.demo-prebuilt-tts.example`

不要上传：

- `.env`
- `.env.local`
- `.env.*.local`
- 任何真实 API key、token、secret
- `node_modules/`
- `dist/`
- `tmp/`
- 本地交付 zip

提交前建议检查：

```bash
git check-ignore -v .env
git ls-files .env
rg -n --hidden --glob '!node_modules/**' --glob '!.git/**' "sk-[A-Za-z0-9]{12,}|(API_KEY|TOKEN|SECRET)=['\"]?[A-Za-z0-9_-]{12,}" .
```

`git ls-files .env` 应该没有输出。若真实 key 曾经提交过，即使后来删了，也要立刻去平台后台 revoke/rotate，因为 Git 历史里仍可能查到。

## 推荐 Git 标识

建议主分支保留完整可开发源码：

```text
main = 完整 LLM 代码 + keyless env 模板
```

演示节点用 tag 标记：

```text
demo-2026-06-08-prebuilt-tts = 路演稳定版，mock 回合 + 预制 TTS
```

不要为了演示复制一整套代码目录；用 `.env` 模板和 tag 区分，会更利于后续继续开发。
